TopicLLM

Coherence-optimized topic modeling with democratic LLM interpretation — running entirely in your browser

This is the working companion to Coherence-Optimized Topic Modeling with Democratic Large Language Model Interpretation: A Human-in-the-Loop Pipeline for Public Health Surveillance of Social Media (Thomas, 2026). It implements the paper's full five-stage pipeline as a client-side web application: your data never leaves your machine except for the LLM calls in Stages 2 and 4, which go directly from your browser to the API endpoint you configure with your own key. There is no server.

1 · Topic discovery

BERTopic-style modeling — MiniLM sentence embeddings, UMAP, HDBSCAN, c-TF-IDF — under randomized hyperparameter search, selecting the model that maximizes cv coherence. Full audit trail.

2 · Democratic naming

Each topic is named by plurality vote over many independent LLM queries: the modal label is adopted, treating naming as estimating a distribution's mode rather than one-shot generation.

3 · Human synthesis

Deliberately not automated. The app builds the two-researcher review packet; named human analysts group topics into themes and record consensus (Theme and Topic system).

4 · Validated classification

Boolean-dictionary and LLM classifiers are piloted against your human-labeled benchmark. The pipeline refuses to scale any classifier below the agreement threshold.

5 · Inference

Proportional z-tests with Bonferroni correction, per-theme corpora for LIWC-22, and the Stata GSEM specification for cross-lagged panel models.

Reproducibility

Every stage emits the artifacts the paper says to publish: hyperparameter logs, vote ballots and tallies, benchmark reports, and test results — downloadable individually or as one archive.

Data & settings

Corpus: a CSV with a tweet (or text) column, plus an optional date column (YYYY-MM-DD) for Stage 5. Benchmark: columns tweet, human_label (the reference implementation uses n=250). Files are parsed locally.

Stage 1 configuration

Reference implementation: 50 iterations × 24 topic solutions = 1,200 candidate models. Browser default is 8 iterations; raise it if you have time. UMAP/HDBSCAN sampling spaces match the paper.

LLM provider (Stages 2 & 4)

Your key is used only for direct calls from this page to the endpoint above and is stored (optionally) in this browser's localStorage. The domain context is the only domain-specific text in the prompts — change it to redeploy the pipeline in a new domain. Pin and report the model identifier; label distributions may shift across model versions.

Stage 2 — Democratic LLM interpretation not run

The reference implementation casts 5,000 votes per topic (temperature 0.5, 15-token maximum). The vote is the measurement — ballots and tallies are downloadable for publication.

Stage 3 — Human-in-the-loop theme synthesis not run

This stage is deliberately not automated: themes emerge through human interpretation under the two-researcher consensus protocol. The primary researcher reads each topic's sample against the domain literature and proposes themes; the senior researcher independently critiques; consensus is recorded here.

Stage 4 — Validated corpus-scale classification not run

Requires the Stage 3 consensus and an uploaded human-labeled benchmark (or the demo benchmark).

How this app relates to the paper

Everything computational in the five-stage pipeline runs client-side: preprocessing, sentence embedding (all-MiniLM-L6-v2 via transformers.js, executed locally with WebAssembly/WebGPU), UMAP (umap-js), HDBSCAN (a JavaScript port with excess-of-mass selection), c-TF-IDF term weighting, cv coherence, plurality-vote tallying, benchmark gating, and the Stage 5 statistics. Stages 2 and 4 send prompts — verbatim from the paper's section 2.7 — directly from your browser to the LLM endpoint you configure.

Fidelity notes (browser vs. Python reference)

  • The Python reference (dtm_pipeline/ in the repository) remains the implementation of record for publication-grade runs; the archival dissertation code is Supplementary Files S1–S3.
  • The WordNet POS-aware lemmatizer is approximated with rule-based inflection stripping.
  • umap-js uses approximate nearest neighbors; results match the reference qualitatively, and the seeded search remains fully auditable via all_iteration_results.json.
  • Equivalent topic solutions are deduplicated: clustering runs once per sampled configuration and is reduced across the topic-count range, rather than refit per count (identical results, far faster).
  • The "hashed" embedder is a no-download fallback for demos — do not use it for real analyses.
  • The "demo mock" provider is not a language model; it exists so the workflow can be exercised end-to-end without credentials. Artifacts it produces are labeled mock-demo.

Practical limits

The browser build is comfortable up to roughly 5,000 documents (embedding is the bottleneck; WebGPU helps). Beyond that, use the Python pipeline. Stage 2's reference budget of 5,000 votes per topic is an API-cost decision — the early-stop control implements the stability-monitoring guidance from the paper.

Privacy

No server exists: your corpus is parsed and analyzed in this tab. The only network traffic is (a) the one-time model download from the Hugging Face Hub, and (b) Stage 2/4 prompts sent to the LLM endpoint you configure with your own key.
